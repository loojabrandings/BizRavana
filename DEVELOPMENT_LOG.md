## Core Development Guidelines

### 1. General Principles
- Build as a scalable multi-tenant SaaS platform.
- Mobile-First responsive design approach.
- Maintain consistent UI/UX across all pages.
- Use reusable components whenever possible.
- Keep the interface simple, clean, and beginner-friendly.
- Optimize for performance and fast loading.
- Use lazy loading where appropriate.
- Follow clean code principles and modular architecture.

### 2. Database Design
- Design for future scalability from day one.
- Every business-related table must include: `business_id`, `created_by`, `created_at`, `updated_at`
- Business account table must include: `plan_id`, `account_status`, `trial_started_at`, `trial_ends_at`, `subscription_started_at`, `subscription_ends_at`, `data_delete_after`
- Never design tables that only work for a single business.

### 3. Multi-Tenant Architecture
- Every user belongs to a Business.
- Every record must belong to a Business.
- Users should never be able to access another business's data.
- Enforce Row Level Security (RLS) in Postgres.
- All data access through Supabase client.
- Use `business_id` for data isolation across all tables.

### 4. Super Admin Architecture
- Super admin status is stored in JWT `app_metadata.is_super_admin`.
- Helper function `is_super_admin()` reads from JWT, not from profiles table (avoids recursion).
- Admin pages at `/admin/*` route group.
- Admin RLS policies check `is_super_admin()` for SELECT/INSERT/UPDATE/DELETE.
- Admin activity logged to `admin_activity_log` table for audit trail.
- Notifications use service-role admin client for server-side operations requiring RLS bypass.

### 5. Notification Architecture
- Notifications are isolated by `business_id` via RLS.
- Shared `NotificationProvider` at the layout level manages one Realtime channel per business.
- Admin broadcasts insert via server-side API route (bypasses RLS INSERT restrictions).
- Automated notifications generated by pg_cron functions (production) or can be triggered manually.
- Notification rules are seed data in migration 024, manageable from admin UI.
- Realtime subscription uses `postgres_changes` with `business_id` filter for efficiency.

### 6. Component Patterns
- Use shadcn/ui primitives from `src/components/ui/`.
- Reusable business components in `src/components/<domain>/`.
- Shared components (DataTable, PageForm, FilterBar) in `src/components/shared/`.
- Admin components in `src/components/admin/` for mobile-responsive patterns.
- Providers in `src/providers/` for cross-cutting concerns.
- Stores in `src/stores/` for Zustand state management.
- Hooks in `src/hooks/` for reusable React logic.

## Entry: Smart Customer Parser
**Date:** 2026-07-22

**Summary:**
Built a Smart Customer Parser that extracts structured customer details (name, phone, address, district, city) from unstructured pasted text — primarily WhatsApp messages, SMS, and social media DMs. Designed for Sri Lankan businesses that receive orders via chat and need to avoid manual re-typing.

### Design Discussion (Before Implementation)
The feature was thoroughly discussed with the user before writing code. Key decisions:
- **Scope**: Customer details only (name, phone, address, district, city) — no item parsing
- **Input examples**: 4 real-world WhatsApp messages with varying formats, missing fields, and Sinhala names typed in English
- **Address rule**: Raw pasted text is preserved as-is — nothing is removed from the address even when fields are extracted
- **Auto-fill**: No preview step — parsing happens in the dialog and immediately fills form fields
- **"Rathnapura" edge case**: Common Sinhala transliteration "Rathnapura" (with 'h') needed to match official "Ratnapura" — solved with transliteration normalization

### Parsing Algorithm
1. **WhatsApp Header Stripping** — Regex removes `[time, date] sender:` prefixes from all lines
2. **Line Classification**:
   - Phone line (07XXXXXXXX pattern) → extracted into phone field
   - First non-phone line → customer name
   - District match (against `SRI_LANKA_DISTRICTS` with Sinhala transliteration normalization) → district field (line still included in address)
   - Everything else → address (unchanged, joined as-is)
3. **City Matching (Two-Phase)**:
   - **Forward Match**: If district is identified, the city dropdown is filtered to cities in that district (from courier data). The last address line is checked against the filtered list.
   - **Reverse Lookup**: If no district found, the last word(s) of the last address line are checked against ALL courier cities. If a match is found, the district is inferred from the city's state_id.
   - Supports multi-word city names (tries full line → last 2 words → last word)

### Files Created
- `src/lib/customer-parser.ts` — Pure parsing utility with 6 exported functions
  - `stripWhatsAppHeaders()` — Strips WhatsApp timestamp headers
  - `isPhone()` / `extractPhone()` — Sri Lankan mobile number detection
  - `matchDistrict()` — District matching with transliteration normalization (`th→t`, `dh→d`, `sh→s`)
  - `matchCityInLine()` — Multi-strategy city matching
  - `lookupCityInAllCities()` — Reverse city-to-district lookup from courier data
  - `parseCustomerText()` — Main orchestrator function

### Files Modified
- `src/components/orders/customer-details-section.tsx`:
  - Added "Smart Parser" button (ClipboardPaste icon + label) in the Customer Details section heading
  - Added paste dialog with: dedicated "Paste from clipboard" button (navigator.clipboard.readText()), manual textarea, accuracy warning banner, "Parse & Fill" button
  - Dialog uses consistent button styles matching the form (variant="outline" for Cancel, variant="gradient" for Parse & Fill)
- `src/components/orders/order-form.tsx`:
  - Added "Reset" button in the footer action bar (before Cancel) that clears all customer detail fields + remarks

### Trace Results (4 User Examples)
| Example | Name | Phone | District | City |
|---------|------|-------|----------|------|
| 1: "Kgdlm pathmanayaka ... Rathnapura" | ✅ `Kgdlm pathmanayaka` | ✅ `0711291725` | ✅ Ratnapura (via normalization) | ✅ Balangoda (forward match) |
| 2: "Tharanga sanjeewa ... 0769325472" | ✅ `Tharanga sanjeewa` | ✅ `0769325472` | ✅ Colombo (via Malabe reverse lookup) | ✅ Malabe |
| 3: "MM Nethmi imesha ... deniyaya" | ✅ `MM Nethmi imesha` | ❌ (no phone) | ❌ (fall through) | ❌ (fall through) |

### Implementation Notes
- Phone detection uses regex to find `07XXXXXXXX` within text (not exact line match), handles embedded phones like "Kamal 0771234567"
- District matching has two paths: exact (fast) then normalized (handles transliteration)
- `normalizeForMatch()` strips 'h' after consonants for common Sinhala variations
- City reverse lookup only works when courier city data is available (courier configured)
- The `ClipboardPaste` icon from lucide-react v1.23.0 is used throughout

## Entry: Global Notification Management System
**Date:** 2026-07-22

**Summary:**
Complete Notification Management System with admin broadcast creation, automated system notifications, Realtime-powered user bell, and shared NotificationProvider.

### Notification Provider
- **Shared context**: Single `NotificationProvider` at dashboard layout level
- **Realtime subscription**: One Supabase Realtime channel per business, filtered by `business_id`
- **Lifecycle**: Fetch on mount → subscribe → refetch on focus → cleanup channel on unmount
- **Events handled**: INSERT (prepend + increment unread), UPDATE (sync is_read/title/message)
- **Optimistic updates**: markAsRead and markAllAsRead update state immediately, then sync to DB
- **No double-decrement**: Realtime UPDATE handler does NOT decrement unreadCount (optimistic update handles it)

### Admin Notifications Page
- **Overview**: Stats cards (total sent, scheduled, drafts, active rules)
- **Create**: Full dialog with title, message, category, priority, audience (all/active/trial/expired/suspended/by plan), action label/URL, send timing (now/schedule/draft), recipient count preview
- **List**: Responsive table showing sent/scheduled/draft broadcasts with status badges
- **Automated Rules**: Switch-toggle enable/disable, essential rule protection, category+priority badges
- **Delivery**: Server-side API route that inserts notifications for each targeted business owner

### Database Migrations
- **024**: notification_broadcasts, notification_recipients, notification_rules tables + RLS + indexes + new columns on existing notifications (source, priority, category, expires_at, action_label/url, broadcast_id) + 14 seed automated rules
- **025**: Automated notification functions (process_auto_notifications, deliver_scheduled_broadcasts, create_business_notification, notification_already_sent) + hourly/15-min cron schedules
